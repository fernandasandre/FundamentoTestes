O que é Teste de Software

  • É verificar se o comportamento verificado no software que está sendo testado é aquele esperado.
  • Verificar se uma determinada entrada reproduz uma saída correspondente.
  • Quando o resultado não é o esperado, estamos diante de uma falha.
  • Testes não verificam todas as saídas posíveis, pois as entradas são infinitas.
  • Testes não garantem qualidade, pois o conceito de qualidade é relativo e depende da percepção do cliente. Qualidade é um conceito mais amplo.
  • Testes custam cerca de 20% a 30% do custo de elaboração do software
  • Corrigir um defeito é mais caro do que evitar que ele ocorra.

O problema atual no mercado

As empresas precisam construir softwares melhores, mais rápido e mais barato. Isso gera:

  • Prazo apertado;
  • Clientes menos tolerantes à falhas;
  • Necessidade de softwares cada vez melhores;
  • Clientes menos tolerantes à atrasos.

As empresas normalmente não estão preparadas para este cenário e acabam entregando seus sistemas com falhas, pois:

  • As tecnologias estão mais complexas;
  • Existem poucos testadores especialistas;
  • As empresas investem pouco em testes;
  • Os softwares acabam sendo testados pelos desenvolvedores, que testam com a visão do sistema perfeito.

Perfil do mercado de testes

  • Poucos testadores;
  • Dos poucos testadores, poucos são especialistas;
  • Preconceito com testes;
  • Difícil encontrar profissionais com perfil, conhecimento e habilidade.

Perfil do testador

  • Detalhista;
  • Paciente;
  • Focado;
  • Criterioso;
  • Perspicaz;
  • Motivado;

O programador tem a visão otimista, enquanto o testador tem a visão pessimista.

O valor do testador é calculado pelo tamanho do problema que resolve versus sua raridade.

Quando testar

Níveis de Teste

Teste de Unidade

É o primeiro teste a ser realizado.

Seu objetivo é testar a menor parte do sistema, como uma classe, uma determinada rotina.

Este teste normalmente é realizado pelo programador, e normalmente são automatizados.

Exemplo de ferramentas utilizadas neste tipo de teste são JUnit, PHPUnit.

Os testes de unidade devem estar sempre atualizados com o código.

Este é um teste de baixo nível, a nível de classes, métodos.

Teste de Integração

O objetivo deste teste é validar a comunicação entre compomentes do sistema.

Estes testes normalmente são realizados por programadores, e focam em encontrar falhas nas interfaces de integração

Normalmente, estes testes são automatizados, feitos antes da conclusão do sistema, e a nível de componente.

Teste de Sistema

Este tipo de teste consiste em executar o software do ponto de vista do usuário final, procurando erros.

São planejados e executados pela equipe de testes, através de cenários de testes com os requisitos especificados.

São realizados após a codificação do sistema, e são à nível de interface.

Teste de Aceitação

São testes realizados do ponto de vista do usuário final, em nível de sistema, e pela interface do usuário.

O teste de aceitação é feito pelo usuário para validar o sistema

Este tipo de teste não procura falhas, apenas usa como homologação, e serve para o cliente aceitar ou não o sistema.

Este teste é a nível de interface.

Teste Alpha

O objetivo deste teste é executar o sistema de forma não planejada, disponibilizando o sistema para um maior número de usuários, que reportam falhas e sugestões de melhoria.

Neste teste, os usuários usam o sistema de forma natural, acompanhados da equipe de desenvolvimento.

Este teste é a nível de interface.

Teste Betha

Este teste consiste em executar o sistema de forma natural, não planejada, e por um grande número de pessoas desconhecidas.

Neste teste, o sistema apresenta uma opção de reportar erros, e é uma espécie de pré-lançamento do sistema, e ao ser pré-lançado, deve informar que é uma versão Betha, para que o usuário saiba que o sistema está em fase de teste.

Teste de Regressão

Pode ser realizado a qualquer momento, e consiste em reexecutar testes já realizados, após alterações no sistema, para verificar se as alterações que foram feitas causaram impacto em outras partes do sistema.

Para este tipo de testes é interessante a implementação de testes automáticos que verificam todas as funcionalidades do sistema.

Estes testes podem ser realizados em qualquer nível do sistema, e servem para reteste.

Como testar

Técnicas de testes

Teste Caixa Branca

Também chamado de caixa aberta, neste tipo de teste, o código do sistema é visualizado por quem está testando. Nele, é possível ver o interior do sistema

São exemplos deste tipo de teste os testes de unidade e teste estático.

  • Teste de cobertura
  • Teste de caminhos
  • Teste de comandos
  • Teste de condições
  • ...

Teste Caixa Preta

No teste estático, o código do sistema não é executado.

Este teste consiste em olhar o código procurando os erros.

  • Teste baseado em cenário
  • Teste baseado em casos de uso
  • Análise de valores limites
  • Teste estático
Itens a conferir no teste estático:
  • Documentação do Código;
  • Loop sem condição de parada;
  • Null pointer não tratado;
  • Variável não inicializada;
  • Se o código obedece a arquitetura pré-definida para o sistema;
  • Tratamento de exceções

Uma ferramenta utilizada para auxiliar neste teste é a find bug.

Teste Dinâmico

São todos os outros tipos de teste que não o estático.

Os testes estático e dinâmico se complementam.

Teste Caixa Branca

Neste tipo de teste, quem está testando não visualiza o código. O teste é realizado com base nas entradas e nas saídas do sistema

São exeplos deste tipo de teste:

  • Teste de integração
  • Teste de sistema
  • Teste de aceitação
  • Teste Alpha
  • Teste Betha

O que testar

Tipos de testes

Teste de funcionalidade

Consiste em validar se as funções do sistema estão operando de acordo com o esperado.

Todos os níveis de testes são úteis para verificar a funcionalidade.

Teste de desempenho

Consiste em verificar o tempo de resposta do sistema para processar uma operação.

Uma ferramenta que pode ser utilizada para implementar este tipo de teste é o JMeter.

Este teste foca em todo o fluxo do sistema - interface, componente, classe - fim a fim.

Teste de usabilidade

Valida a experiência do usuário com o sistema, tais como cores, layout, facilidade de uso.

O foco deste tipo de teste não é encontrar defeitos, mas sim no prazer que o usuário sente em manipular o sistema.

Teste de segurança

Validam a proteção do sistema contra invasões e acessos não autorizados, e levam em consideração aspectos como criptografia e controle de acesso.

Consiste em o testador "hackear" o sistema, procurando vulnerabilidades, antes que um hacker mal intencionado o faça.

Teste de portabilidade

Validam o funcionamento do sistema em diversas plataformas, nas quais ele se propõe a funcionar.

Podem ser verificados neste tipo de teste a diversidade de browsers.

Teste de stress

Este tipo de teste valida o sistema em situações extremas, como por exemplo, testar o sistema com milhares de acessos, ou milhares de usuários simultaneamente.

Uma ferramenta que pode ser utilizada neste tipo de teste é o JMeter.
